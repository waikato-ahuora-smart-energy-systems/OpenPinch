This matches the migration plan's LMTD helper-replacement lane and does not take the separate `pinch_classes`, costing, or export-helper lanes (`OPENHENS_MIGRATION_PLAN.md:232-238`, `OPENHENS_MIGRATION_PLAN.md:1038-1047`).
- The implementation keeps the OpenHENS active-unit and dTmin/tolerance gates in the synthesis model layer, then delegates only the positive-endpoint logarithmic-mean formula to the shared OpenPinch helper (`OpenPinch/services/heat_exchanger_network_synthesis/models/base.py:120-138`). The shared OpenPinch helper still owns positive endpoint validation, near-equal endpoint handling, and the logarithmic denominator (`OpenPinch/utils/heat_exchanger.py:27-49`).
- I did not see any equation model move, new algorithm, solver default change, tolerance widening, public API expansion, or workflow bypass in the HENS-10 diff. The existing public-path guard coverage still rejects target/root service bypasses and raw-payload service calls (`tests/test_heat_exchanger_network_synthesis_workflow.py:225-237`).

- Structural and export evidence is sufficient for this non-solver-moving helper slice. The task records the focused helper command, focused HEN/export/topology command, fast non-solver suite, ruff, full ruff, and diff whitespace checks (`docs/developer/openhens-integration-tasks/10-duplicate-helper-replacement.md:130-146`). - Documentation and comments are library-quality for this slice. The new helper docstring states the stable ownership contract and domain invariant without claiming a new scientific convention (`OpenPinch/services/heat_exchanger_network_synthesis/models/base.py:129-134`), and the task notes distinguish source provenance, ownership, command evidence, and solver blockers without hiding non-equivalence or implying an OpenHENS compatibility surface (`docs/developer/openhens-integration-tasks/10-duplicate-helper-replacement.md:109-153`).

## Residual Risks

- I did not rerun pytest, ruff, or the intentional helper-drift mutation because this review is read-only apart from this review artifact; those commands can write caches or generated artifacts. I relied on the recorded command results plus direct inspection of the code and tests.
- Live moved OpenPinch Four-stream solver parity has not run in this environment. It should be rerun when `couenne` and `ipopt` are available, or when a marked OpenPinch solver regression tier is added.
- The current Four-stream structural proof replays a saved source artifact through OpenPinch adapter/extraction code; it does not prove the moved OpenPinch solver path can regenerate that artifact.
- `.DS_Store` remains modified in the worktree and outside HENS-10 scope. It should stay excluded from staging/publication.

## Verdict

HENS-10 is cleared for this review pass. The remaining solver gap is an explicitly documented environmental/marker residual risk, not a current review-blocking finding.
